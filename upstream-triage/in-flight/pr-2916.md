# upstream PR #2916 — Add `--layout` to workspace.create for programmatic split layouts

**Evaluate:** maybe — semantic collision with c11's blueprint system on the same flag name and same socket param name
**Difficulty:** moderate (touches `CLI/c11.swift` rename + `Sources/TerminalController.swift` adapt zone)
**Approach:** rewrite (cherry-pick fails on the rename pattern + real semantic conflict)

## What this PR does

Adds a `layout` param to the `workspace.create` socket method and a `--layout <json>` flag to `cmux new-workspace`. The layout JSON is decoded as `CmuxLayoutNode` (the same schema `cmux.json` custom commands already use) and routed through `Workspace.applyCustomLayout()`. Lets the operator/agent spin up a workspace with predefined splits in a single call instead of `create` + a series of `split`/`send` calls.

Files: `CLI/cmux.swift` (+20 −8), `Sources/TerminalController.swift` (+20 −2), `tests_v2/test_workspace_create_layout.py` (+328 new). Merge commit `11ac1fa9`, author @talldan.

## What c11 already has

c11 already accepts `layout` on `workspace.create` and `--layout` on `c11 new-workspace`, but with a different shape — c11's CMUX-37 **blueprint** system:

- **Socket `workspace.create`**: `params["layout"]` is a *blueprint plan dict*, dispatched to `v2WorkspaceApply` (`Sources/TerminalController.swift:3726`). Returns workspace_ref + layout_result. The plan is the apply-plan structure that `c11 workspace apply` consumes.
- **CLI `c11 new-workspace --layout <path|name>`**: resolves a blueprint by file path or named blueprint via `resolveBlueprintPlan(...)`, then sends the resolved plan as `params["layout"]` (`CLI/c11.swift`, around line 1841).
- **Adjacent c11 commands**: `c11 workspace new`, `c11 workspace apply`, `c11 workspace export-blueprint`, `c11 snapshot`, `c11 restore`, `c11 list-snapshots`. The CMUX-37 surface area covers programmatic layout creation already.

Upstream's `layout` is a `CmuxLayoutNode` tree — a recursive {direction, split, children, pane{surfaces[]}} structure. c11's `layout` is a blueprint *plan* — a richer object that names the layout, threads per-surface env vars, snapshot identities, and the workspace.apply state machine.

## The collision

Same flag name, same socket param name, **incompatible schemas**. Upstream's `--layout '{"direction":"horizontal",...}'` would fail c11's `resolveBlueprintPlan` (it expects a path or blueprint name). Upstream's `params["layout"]: CmuxLayoutNode` would fail c11's `v2WorkspaceApply` (it expects a blueprint plan dict).

Probe confirmed it: `STATUS=conflict` in `CLI/c11.swift` and `Sources/TerminalController.swift`. Real semantic conflict, not just rename mechanics.

## How I'd land it on c11 (if Atin says yes)

Three options, ranked by fit:

1. **Skip — c11's blueprint system already covers the use case.** The intent ("agent spins up a workspace with predefined splits in one call") is already achievable today via `c11 workspace new --blueprint <name>` or by passing a blueprint plan to `workspace.create`. The upstream PR is upstream catching up to where c11 already is. Cost: zero. Recommendation if the goal is "minimize divergence ceremony".

2. **Coexistence — accept either shape on `--layout` / `params["layout"]`.** Detect at parse time: if the value parses as a JSON object with `direction`/`split`/`children`/`pane` keys → treat as upstream `CmuxLayoutNode` and route through `applyCustomLayout`. Otherwise → existing blueprint path. Touches both files, ~50–80 LOC, plus port the upstream test (`tests_v2/test_workspace_create_layout.py`). Preserves backward compatibility on both shapes. Cost: real but bounded.

3. **Rename upstream's flag — `--layout-json` / `params["layout_json"]`.** Keeps both schemas alive without ambiguity. Easier to reason about, but introduces a c11-specific divergence on the flag name. The test file would need adapting too.

I'd recommend (1) skip — the upstream PR's value is "this capability exists", and c11's blueprint system is strictly more capable on this surface. Carrying the upstream form would just be doubled API for the same effect.

## Open questions

- Is anyone in c11's user base specifically asking for the upstream `CmuxLayoutNode` shape? (If c11's blueprint system covers everything, the answer is probably no.)
- Does the `tests_v2/test_workspace_create_layout.py` test file have any value to import as-is — i.e. coverage gaps in c11's existing blueprint tests?
- If we go with (2) coexistence, does anyone want the `--name` and `--description` flags upstream is also adding back to `new-workspace`? c11 dropped those at some point; might be worth re-adding for parity if they're still useful.

## Action

**Atin nodded on coexistence (no sweeping rename).** Locked-in approach:

- `--layout <value>` becomes a superset that dispatches at parse time (in this priority):
  1. Inline JSON (starts with `{`, parses as object with upstream `CmuxLayoutNode` shape — `direction`/`split`/`children`/`pane` keys) → route through `applyCustomLayout`.
  2. Path on disk → load blueprint envelope, unwrap plan, route through `v2WorkspaceApply` (existing c11 path).
  3. Registered blueprint name → look up, unwrap plan, route through `v2WorkspaceApply` (existing c11 path).
- Same dispatch on the socket: `params["layout"]` sniffs shape and routes. Fully backward compatible.
- The Blueprint concept stays unchanged — it remains the named persisted envelope wrapping a `WorkspaceApplyPlan`.
- Port the upstream test file (`tests_v2/test_workspace_create_layout.py`).
- Optional surface-alignment renames (`list-blueprints` → `list-layouts`, `export-blueprint` → `export-layout`) deferred to a separate PR if/when desired.
